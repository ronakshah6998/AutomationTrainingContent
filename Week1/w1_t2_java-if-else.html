If-Else Conditions in Java

Control which code runs and when — from simple checks to multi-branch ladders used in every real automation workflow.

🔀 Decision Logic ⚙ Automation Ready ✅ Branching ⚡ Interactive
What is Decision Making in Programming?

Real programs don't run the same path every time. Conditions let your code choose what to do next based on data — this is the engine behind all automation logic.

💡
Real-world analogy: Think of a traffic light. If it's Green → you drive. Else if it's Yellow → slow down. Else → stop. Java's if / else if / else routes your program the same way — only the matching branch runs, all others are skipped.
🔀

Decision Making

if-else lets your program act differently depending on whether a condition is true or false at runtime.

🌿

Branching Paths

Code splits into branches. Only the first matching branch executes — all others are completely skipped by the JVM.

🔗

Chained Conditions

The else if ladder checks multiple conditions in sequence. First true match wins — the rest are ignored.

🛡

Safe Fallback

The final else block is your default. It only runs when no condition above was true. Never skip it in production code.

How Automation Uses If-Else

In every automation framework, conditional logic drives these three core patterns:

  • 1

    Validation

    Check whether inputs are valid before proceeding. Example: if (field.isEmpty()) { flagError(); }

  • 2

    Branching

    Navigate different flows based on state. Example: if (isLoggedIn) { goToDashboard(); } else { redirectToLogin(); }

  • 3

    Result Handling

    Evaluate responses and mark test outcomes. Example: if (statusCode == 200) { pass(); } else { logFailure(); }

🚀
Pro tip: In Selenium, REST Assured and TestNG — 90% of your assertion logic uses if-else directly, or through assert methods that wrap it internally. Nail this now and your test code will be readable from day one.
Syntax Anatomy

Every if statement has the same structure. Understand this once — it covers all three forms.

if Keyword
 ( marks > 90 )  Boolean Condition
 { ... }  Block (runs if true)
else { ... } Fallback (runs if false)
Form 1 — Simple if

Runs a block only when the condition is true. Skips it silently when false.

☕ Simple if Statement
int score = 85;

if (score >= 90) {
    // runs only when score is 90 or above
    System.out.println("Excellent!");
}
// score = 85 → condition is FALSE → nothing prints
Form 2 — if-else

Two-way branch. One path runs when true, the other when false. Always exactly one executes.

☕ if-else Statement
int score = 45;

if (score >= 50) {
    System.out.println("PASSED ✅");
} else {
    // runs when the if-condition is false
    System.out.println("FAILED ❌");
}
// Output: FAILED ❌  (45 < 50)
Form 3 — if-else-if Ladder

Multiple mutually exclusive branches. Java evaluates top-to-bottom and stops at the first true match.

☕ if-else-if Ladder
int marks = 75;

if (marks > 90) {
    System.out.println("Grade A — Distinction");
} else if (marks > 60) {     // ← only checked if above was false
    System.out.println("Grade B — Pass");
} else {                          // ← fallback for everything else
    System.out.println("Grade C — Fail");
}

// marks=75: (75>90)=false → (75>60)=TRUE → "Grade B — Pass"

Common If-Else Mistakes

These errors come up constantly — in beginner code and real automation projects alike. Learn to spot and fix them.

🔴 Error: Using assignment = instead of comparison ==

What went wrong?

A single = assigns a value — it does not compare. Java throws a compile error when you use it inside an if condition for primitives, but for booleans it silently always evaluates to true.

error: incompatible types: int cannot be converted to boolean if (statusCode = 200) ^
❌ Wrong
int statusCode = 200; if (statusCode = 200) { // ❌ assignment! System.out.println("OK"); }
✅ Correct
int statusCode = 200; if (statusCode == 200) { // ✅ comparison System.out.println("OK"); }
✅ Rule: Use == to compare. Use = only to assign. They are completely different operations.
🔴 Error: Missing curly braces { }

What went wrong?

Without braces, only the very next single line belongs to the if. Every line after it runs unconditionally — a silent bug that is extremely hard to trace.

// Bug: "Log error" always runs, even when response != 404 if (response == 404) System.out.println("Not Found"); // in the if System.out.println("Log error"); // NOT in the if! ❌
❌ Wrong
int response = 200; if (response == 404) System.out.println("Not Found"); System.out.println("Log error"); // always runs!
✅ Correct
int response = 200; if (response == 404) { System.out.println("Not Found"); System.out.println("Log error"); // inside { } ✅ }
✅ Rule: Always use { } even for a single-line block. Add a line later and you won't introduce a bug.
🔴 Error: Broad condition placed before specific condition

What went wrong?

Java evaluates top-to-bottom and stops at the first match. A broad condition like score > 50 will catch values that should have matched a narrower condition — making the narrower condition dead code.

// Bug: score=85 hits "score > 50" first → prints "Pass" // "Merit" and "Distinction" blocks can NEVER be reached
❌ Wrong — broad first
int score = 85; if (score > 50) { // catches 85 → STOPS System.out.println("Pass"); } else if (score > 80) { // dead code ❌ System.out.println("Merit"); } else if (score > 90) { // dead code ❌ System.out.println("Distinction"); }
✅ Correct — specific first
int score = 85; if (score > 90) { // most specific System.out.println("Distinction"); } else if (score > 80) { // 85 lands here ✅ System.out.println("Merit"); } else if (score > 50) { System.out.println("Pass"); }
✅ Rule: Always order from most specific to most general. Narrow conditions first.
🔴 Error: Logical mistakes — wrong operator or inverted logic

What went wrong?

Using == to compare String objects checks memory addresses, not content. And accidentally inverting a condition with or without ! causes the opposite branch to execute.

// Bug 1: == on String compares references, not text // "admin" literals may or may not share the same reference in memory // Bug 2: Missing ! means admins are denied instead of non-admins
❌ Wrong
String role = "admin"; if (role == "admin") { // ❌ reference compare grantAccess(); } if (role.equals("admin")) { // ❌ blocks admins! System.out.println("Access denied"); }
✅ Correct
String role = "admin"; if (role.equals("admin")) { // ✅ content compare grantAccess(); } if (!role.equals("admin")) { // ✅ denies non-admins System.out.println("Access denied"); }
✅ Rule: Always use .equals() for String comparison. Use ! to negate a condition cleanly.
